## Objetivo
Fechar os 4 gaps reais + demografia na Ficha + agregação de familiares:
1. Depoentes do dossiê → pessoas (adapter novo no `promocao/`).
2. Geocode de residências (depoentes/assistido) via pipeline `lugares` existente.
3. Familiares do assistido (réu) — **1 tabela nova** + backfill de texto livre.
4. Ficha da Pessoa 360° — demografia (idade/telefone/endereço) p/ réu/vítima/depoentes + familiares + mapa.
5. (Contingente) embeddings ↔ pessoa — substrato Caseload-RAG.

Princípio: TDD-first (núcleo puro testado antes), idempotente, em-groove com o padrão
`adaptador-*/planejar/applier/repo` + hook `consolidateForProcesso`.

## Fase A — Depoentes → pessoas

(ACUSACAO/DEFESA), `telefone`, `endereco`, `ouvidoEm`, `resumoDepoimento`. Mais
estruturada que `analysisData.depoimentos[]` e já é o que a sheet usa; permite ligar
`participacoesProcesso.testemunhaId` e alimentar Fase B (endereço) e Ficha (telefone).
> Nota: `analysisData.pessoas[]` já é promovido por `adaptador-analysis`. O resolver
> deduplica por nome+nascimento/cpf, então um depoente que já virou pessoa **vincula**
> (não duplica). Idempotência reforçada por `fonteRef="depoentes:{processoId}"`.

**Arquivos:**
- CRIAR `src/lib/promocao/adaptador-depoentes.ts` → `candidatosDeDepoentes(processoId, rows: Testemunha[]): CandidatoPessoa[]` (filtra com nome; mapeia tipo→papel via `de-para-papeis`; `confianca` 0.8; carrega `testemunhaId` em metadado p/ o repo ligar).
- EDITAR `src/lib/promocao/repo.ts` + `tipos.ts` → permitir `testemunhaId` opcional no candidato e setá-lo em `inserirParticipacao` (hoje não seta).
- EDITAR `src/lib/promocao/backfill.ts` (`promoverProcesso`) → carregar `testemunhas` do processo e concatenar `candidatosDeDepoentes(...)` aos candidatos.
- EDITAR `src/lib/services/intelligence-consolidation.ts` → já chama `promoverProcesso` (que passa a incluir depoentes); nenhuma chamada nova necessária.
- (Opcional) tRPC `promocao.backfillDepoentes` seguindo o padrão dos outros backfills.

**TDD:** `adaptador-depoentes.test.ts` (filtro, mapa de papel, fonteRef, confiança);

## Fase C — Familiares (ÚNICA tabela nova)
**Por quê nova:** hoje familiares são texto livre (`assistidos.nomeMae`/`nomePai`/
`nomeContato`/`parentescoContato`; `atendimentos_triagem.familiarNome`/`familiarGrau`).
Não há aresta pessoa↔pessoa estruturada. O usuário quer agregar familiares do réu.

**Schema novo** em `src/lib/db/schema/pessoas.ts` (aditivo):
```
pessoa_relacoes
  id serial PK
  pessoa_id            int NOT NULL FK pessoas        -- o assistido/réu
  relacionada_pessoa_id int FK pessoas (nullable)     -- se o familiar já é pessoa
  grau                 varchar(40)                     -- mae|pai|conjuge|filho|irmao|contato|outro
  nome_livre           text                            -- fallback quando não há pessoa
  telefone             varchar(20)
  endereco             text
  fonte                varchar(40)                     -- backfill-assistido|triagem|manual
  fonte_ref            varchar(120)                    -- idempotência
  confirmado           boolean default false
  created_at/updated_at
  UNIQUE (pessoa_id, grau, nome_livre)  -- evita duplicar no backfill
```
**Backfill** `src/lib/promocao/backfill-familiares.ts`: para cada assistido com
`pessoa_id`, cria relações a partir de nomeMae(mae)/nomePai(pai)/nomeContato(grau=
parentescoContato). Tenta `relacionada_pessoa_id` por match em `pessoas` (resolver
existente); senão usa `nome_livre`. Idempotente por `fonte_ref`.
**tRPC** `pessoas.getFamiliares`/`addFamiliar`/`removeFamiliar`.

**TDD:** mapeamento texto-livre→relação; idempotência; match opcional a pessoa.

## Ordem e dependências
```
A (depoentes→pessoa)  ─┐
B (geocode resid.)     ─┼─> D (Ficha 360°)
C (familiares: tabela) ─┘
E (embeddings↔pessoa)  ── verificar → estende D ("menções")
```
Sugestão: **A → B → C → D → E**. A/B/C produzem dados; D os consome; E por último.

## Riscos / salvaguardas
- Não duplicar pessoas: confiar no resolver + `fonteRef`; testes de idempotência.
- Não quebrar a sheet: Fase A lê `testemunhas` (não reescreve JSON).
- `pessoa_relacoes` é a única migração; aditiva, com down-migration.
- Migrações são **hand-authored** neste repo (journal congelado) — seguir formato dos `00xx_*.sql`, não usar `db:generate` interativo.
