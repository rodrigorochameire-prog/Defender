## Contexto

A Fase 1 introduziu o modelo de Drive multi-tenant:

- Tabela `drive_groups` com o jsonb `atribuicaoFolders` (`Record<string, string[]>`: atribuição → lista de pastas), `ownerUserId`, e `users.driveGroupId`.
- Resolver `userId`-first em `src/lib/services/drive-folders.ts`:

  - `resolveAllAtribuicaoFolders(userId)` → mapa completo do grupo.

A Fase 1 **deferiu deliberadamente** o `google-drive.ts` para a Fase 1.5 (commits: *"ATRIBUICAO_FOLDER_IDS vira grupo padrao legado (google-drive.ts -> Fase 1.5)"*). A constante global `ATRIBUICAO_FOLDER_IDS` (pastas do Rodrigo, 9ª DP) foi rebatizada como **"grupo padrão legado"** justamente para servir de fallback nesta fase.

## Escopo real medido no código

Restam **4 clusters** em `src/lib/services/google-drive.ts`:

| Local | Tipo | Função | Hoje |
|---|---|---|---|
| ~L3502 | forward | `createAssistidoFolder` (site interno) | `ATRIBUICAO_FOLDER_IDS[atribuicao]` |
| ~L3533/34 | forward | `moveAssistidoFolder(folderId, old, new)` | `ATRIBUICAO_FOLDER_IDS[old]` / `[new]` |
| ~L3887 | forward | `listAssistidoFoldersWithCount(atribuicao)` | `ATRIBUICAO_FOLDER_IDS[atribuicao]` |
| ~L4222/4260/4419 | reverse | `FOLDER_ID_TO_ATRIBUICAO`, `handleNewAssistidoFolder`, `isAtribuicaoRootChild` | mapa global Rodrigo-only |

> A estimativa do handoff ("~12 usos") era alta: a Fase 1 já migrou os demais arquivos. `SPECIAL_FOLDER_IDS` (DISTRIBUICAO etc.) **não é tocado** — é usado por webhooks/upload/whatsapp/inngest e independe de grupo.

Nenhuma das funções forward recebe `userId` hoje — o threading de propriedade (ownership) é a principal mudança de assinatura.

## Decisões de design (aprovadas)

1. **Fonte da resolução reversa:** consultar `drive_groups.atribuicaoFolders` (a mesma fonte única de verdade do resolver forward). Rejeitados: `drive_sync_folders` (não carrega atribuição) e tabela de índice reverso materializada (YAGNI nesta escala).
2. **Profundidade do multi-tenant na reverse-sync:** resolver o **dono** e **escopar downstream** — assistido criado/vinculado recebe `defensorId = ownerUserId`. O match de deduplicação permanece **global** (modelo de assistidos COMPARTILHADOS de `defensor-scope.ts`), evitando registros de pessoa duplicados entre defensores.
3. **Fallback legado:** quando o grupo não resolve (usuário sem `driveGroupId`), cair para `ATRIBUICAO_FOLDER_IDS[atribuicao]`. Preserva o comportamento single-tenant atual até a infra (seed/onboarding) rodar.

## Arquitetura

### Unidade 1 — Resolução reversa (`drive-folders.ts`)

Par puro + async, espelhando os resolvers forward existentes:

```ts
/** Pura: varre o mapa de um grupo procurando o folderId → chave de atribuição. */
export function findAtribuicaoForFolder(
  folders: AtribuicaoFoldersMap,
  folderId: string,
): Atribuicao | null;

/** Async: lookup reverso entre todos os grupos, sem sessão. */
export async function resolveFolderToAtribuicao(
  folderId: string,
): Promise<{ ownerUserId: number; driveGroupId: number; atribuicao: Atribuicao } | null>;
```

- `findAtribuicaoForFolder` reaproveita a coerção Array de `pickAtribuicaoFolders` (tolera valor string legado); itera as chaves do mapa e retorna a primeira atribuição cuja lista contém o `folderId`.
- `resolveFolderToAtribuicao` faz `SELECT id, ownerUserId, atribuicaoFolders FROM drive_groups` (1–dezenas de linhas) e varre em JS via `findAtribuicaoForFolder`. Retorna `null` se nenhum grupo contém o folder.
- **Otimização futura documentada (não implementada):** índice GIN / `jsonb_path_exists` quando o número de grupos crescer. YAGNI agora.

### Unidade 2 — Fallback legado (helper local em `google-drive.ts`)

```ts
// Resolver por grupo, ou constante legada se o usuário ainda não tem grupo.
async function folderForAtribuicaoOrLegacy(
  userId: number | null,
  atribuicao: Atribuicao,
): Promise<string | null>;
```

Retorna `resolveAtribuicaoFolder(userId, atribuicao)` quando há resultado; senão `ATRIBUICAO_FOLDER_IDS[atribuicao] ?? null`.

### Unidade 3 — Reverse-sync (webhook)

- `handleNewAssistidoFolder(folderId, folderName, parentFolderId)`:
  - troca `FOLDER_ID_TO_ATRIBUICAO[parentFolderId]` por `resolveFolderToAtribuicao(parentFolderId)`; `null` → skip (contrato inalterado).
  - na criação/vínculo do assistido, grava `defensorId = ownerUserId`.
  - match fuzzy permanece global (não filtra por defensor).
- `isAtribuicaoRootChild(parentDriveId)` vira `async` → `(await resolveFolderToAtribuicao(parentDriveId)) !== null`. O único chamador (`syncIncremental`) passa a `await`.
- `FOLDER_ID_TO_ATRIBUICAO` é removido.

### Unidade 4 — De-hardcode forward (3 sites)

Threading explícito de `ownerUserId`, com fallback legado:

| Função | Grupo de quem | Resolução |
|---|---|---|
| `createAssistidoFolder` (~L3502) | defensor responsável do assistido | `folderForAtribuicaoOrLegacy(assistido.defensorId, atrib)` |
| `moveAssistidoFolder` | defensor do assistido | novo parâmetro `ownerUserId`; resolve pasta antiga e nova |
| `listAssistidoFoldersWithCount` | usuário em ação | novo parâmetro `ownerUserId` |

Os chamadores (routers tRPC) passam `assistido.defensorId ?? ctx.user.id` (mover/criar) ou `ctx.user.id` (listar).

## Fluxo de dados (reverse-sync multi-tenant)

```
Webhook Drive (folder novo em raiz de atribuição)
  → handleNewAssistidoFolder(folderId, nome, parentFolderId)
     → resolveFolderToAtribuicao(parentFolderId)
        → SELECT drive_groups → findAtribuicaoForFolder → {ownerUserId, atribuicao} | null
     → null? skip
     → match fuzzy global em assistidos
     → link/cria assistido com defensorId = ownerUserId
     → agenda enrichment
```

## Tratamento de erros

- Lookup reverso nunca lança: grupo ausente / mapa vazio / valor legado string → `null`.
- Fallback legado cobre usuário sem `driveGroupId` (single-tenant preservado).
- `SPECIAL_FOLDER_IDS` intocado.
